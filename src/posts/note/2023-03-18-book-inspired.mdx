### 기억에 남는 글
- 만들 만한 가치가 있는 제품이 아니라면 엔지니어팀이 얼마나 훌륭한 지는 아무 의미가 없다.
- 실패한 제품의 근본 원인 (= waterfall)
    - 아이디어 -> 로드맵 -> 요구사항 -> 디자인 -> 구현 -> 테스트 -> 배포
    - 왜? 중요한 일을 먼저 수행해 주기를 원하며, 언제 준비될지 예측하고 싶기 때문에.
    - 제품 관리자 : 요구 사항 수립
    - 디자이너 : 요구 사항 디자인
    - 엔지니어 : 구현 시작 (애자일, 스프린트, QA, 배포)
    - 문제점
        - 아이디어의 원천 : 이해 관계자 위주로 제품이 끌려가게 된다.
        - 정말 뛰어난 아이디어를 가져 오지는 못할 것, 아이디어의 최소 2/3은 쓸모 없음
        - 권한 위임이 안됨. 제품팀은 그냥 원하는 바를 구현 해 주는 용병. 주인 의식 X
        - 제품 개발 -> 모르는 것을 알아가는 과정. 현재 시점에서 얼마만큼 돈을 벌수 있고, 얼마나 비용이 드는가 계산 할 수 없음 : 솔루션의 퀄리티에 달려있을 뿐.
        - 실제로 구현을 한다고 해도 현재 해결하고자 하는 미션을 해결할 수 있을지는 알수 없음.
        - 정말 뛰어난 아이디어라고 해도, 수익 창출 과정까지는 몇번의 이터레이션 과정을 반복해야 한다. (TTM)
        - 해당 모델에서의 PM : 요구사항을 수집하고, 문서화만 하는 역할
        - 디자이너, 개발자 : 이렇게 하는것이 바람직하지 않다는걸 알고 있지만, 현재 상황에서 할수 있는 방식으로 그냥 진행 하게 됨.
        - 엔지니어들이 너무 늦게 참여해서, 혁신을 하는데 가장 훌륭한 원천을 놓치고 있는 것
        - 애자일 원칙과 핵심의 장점이 너무 늦게 작동한다. (제품 구현 단계에서만 애자일 적용.)
        - 프로젝트 중심적, 문제 상황을 해결하는데 집중하지 않는다. 프로젝트의 성공과 문제 상황에 대한 해결은 다르다.
- 제품 팀을 만들면 어떤 점이 좋아지는가?
    - 협업에서는 관계가 중요한 역할을 하는데, 제품 팀은 이를 잘 살리는 형태
    - 전문성이 충분히 확보될 만큼 깊이 업무 가능
    - 팀 전체가 주인의식과 성과에 대한 책임감을 느낀다.
    - 기능 구현으로 끝이 아니라, 원하는 미션이 달성될 때 까지 책임감을 느낀다.
- 제품 팀의 권한과 위치
    - 같은 장소에서 근무하면 효율이 더 높다.
    - 업무 범위 : 제품과 관련된 모든 책임
    - 일의 범위 : 회사에서 필요에 따라 설정
    - 팀의 지속성 : 프로젝트를 수행하기 위해 형성된 조직이 아님. 주인 의식을 느끼기 위해서는 충분한 지속성 필요
    - 팀의 자율성 : 팀이 판단 하기에 주어진 문제를 해결 하는데 최고의 방법 시도


- CTO로서의 역할 (104p)
    - 조직 : 직원들의 역량 개발에 대해 신경 쓰고, 좋은 팀 조직을 구성 하는 일
    - 리더십 : 비즈니스 전략 방향, 기술적 조언, 기술 대표
    - 제품 실행 : 신속하고, 믿을 수 있는 품질의 제품을 시장 제공 가능 하도록 관리 하는 책임
    - 아키텍쳐 : 전체를 바라보고, 인프라가 비즈니스 성장 속도에 맞춰지고 있는가?
    - 제품 발견 : 엔지니어들이 제품 발견에 참여 하는 지속성과 범위를 관리
    - 기술 전도사 : 기술 조직의 대변인, 협력사, 고객 커뮤니티에 리더십 발휘

- 제품 팀을 나누는 원칙 (111p)
    - 마법같은 원칙은 없다
    - 투자 전략과 연계하여 : 제 역할을 못하는 팀을 줄이고. 캐시 카우 제품에 투자를 줄이며 새로운 매출과 성장에 집중
    - 상호 의존의 최소화 : 더 빨리, 더 많은 자율성을 느낄 수 있도록
    - 주인 의식과 자율성 : 용병이 아닌 미션팀을 원한다. 충분한 권한을 가지고 제품에서 중요한 부분을 책임지고 있다는 것을 느낄수 있게.
    - 레버리지 극대화 : 공통적인 요청, 공유하는 서비스에 대해 대응하기 위한 새로운 공유 서비스 -> 사내툴
    - 제품 비전과 전략 : 우리 조직이 도달해야 할 곳이 어디인지 설명과 설정, 이걸 잘 실행하기 유리한 형태로 구성
    - 팀의 규모 : 한 팀에 한명의 제품 관리자, 디자이너와 엔지니어 2명이 최소 단위
    - 아키텍쳐와 연계 : 비전을 실행하기 위한 아키텍쳐 구상, 그리고 아키텍쳐를 중심으로 설계. 다른 팀에 공통 서비스를 제공하는 팀 (플랫폼 팀)
    - 사용자 또는 고객과 연계 : 담당하는 고객에 대해 깊이 이해할 수 있기 때문에 유리.
    - 비즈니스와 연계 : 여러개의 사업 영역에 대해 고려. 보통은 여러개의 사업이 공통적, 통합적 기반으로 만들어 졌으므로 대체로 다른 기준을 먼저 반영한 후 고려
    - 구조는 움직이는 목표물이다 : 팀을 구성하는 완벽한 방법은 없음. 시간이 지남이 따라 변해야만 하고, 1년에 한번 정도 변하는게 일반적이다.

- 제품 로드맵의 문제 (134p)
    - 최소 절반 이상의 아이디어는 유효하지 않을 것.
    - 지나치게 복잡해서 쓰임새보단 골칫거리가 많을 것.
    - 제품 생산시 예상보다 더 많은 자원이 필요해 실현 가능성이 없는 것.
    - 법적, 재무적 제약 사항이 출시를 막을 수도 있음.
    - 유효성이 검증되었다고 하더라도, 비즈니스 가치를 만들기까지 최소 몇번의 이터레이션을 반복 해야 함.


- 제품 비전과 제품 전략, 제품 원칙 (145p)
    - 제품 비전 : 2-5년 정도의 기간에 우리가 만드는 미래.
    - 제품 전략 : 비전을 실현하기 위한 과정
    - 제품 원칙 : 만들고자 하는 제품의 특성 (ex, 구매자와 판매자가 원하는 것이 충돌할 경우, 구매자가 필요로 하는것에 우선한다.)
    - 제품 비전을 가진다는건 어느정도 믿음을 가진다는 것.
    - 제품 비전은 영감을 줄 수 있어야 하고, 제품 전략은 분명한 초점이 있어야 한다.

- 어디에 시간을 쓸 것인가 (166p)
    - 엔지니어가 플랫폼을 개선하고, 디자이너는 반응형 디자인으로 전환 하는 데 쓰는 시간은 비즈니스 문제 해결에 비해 우선순위가 높지 않다.
    - 더 큰 상위의 목표를 가지고 있다면, 다른 사업 목표와 연계하여 리더십 팀 레벨에서 먼저 우선순위를 논의하라.
    - 각 개인이 그들의 제품팀의 목표에 주목할 수 있도록 하라.

- 열정을 보여주는 방법을 배워라 (175p)
    - 절대 진실하되, 당신이 정말로 설레고 있음을 사람들이 확인 할 수 있게 해 주어라. 열정은 정말로 전염성이 있다.

### 느낀점

1. 좋은 개발조직에서는 팀 전체의 목표가 `구현 완료`에 맞춰져 있지 않고, `미션 해결`에 맞춰져 있다는 사실을 배웠다.
나는 특정 기능이 필요한 경우에 특정 기능을 구현 하는데만 집중 했었고, 특정 기능을 통해 무엇을 개발 하고자 하는지, 그 방식이 그 문제를 해결하는데 최선일지는 생각해보지 않았다.
앞으로는 스프린트 미팅과 킥오프 미팅에서 정말로 우리가 해결하고자 하는 문제가 무엇인지, 그래서 그 기능 개발을 통해 문제를 해결할 수 있는지 좀 더 적극적으로 파악하려고 해야겠다.

2. 최고의 아이디어는 결국 엔지니어에게서 나오는 경우가 많다고 한다. 엔지니어가 제품 발견, 기획 단계에서 적극적으로 참여 할 수 있도록 `핵데이`와같은 장치들을 많이 마련해야 겠다는 생각을 했다.

3. 좋은 제품팀과 나쁜 제품팀을 비교한 자료를 보니, 현재 우리팀은 좋은 제품팀의 기준에 부합하는 항목도 있지만 나쁜 제품 팀의 기준에 부합하는 내용도 많았다. 해당 내용들을 정리해서
동료들과 공유 해 보고, 지속적으로 전파해 우리 팀이 좋은 제품 팀으로 거듭날 수 있도록 지속적으로 알리는 노력을 해야겠다.

4. 강력한 혁신 문화와 실행 문화를 가지는 기업을 만드는 일은 정말로 쉽지 않다. 그리고 그 중심에는 뛰어난 제품 관리자가 있다.
