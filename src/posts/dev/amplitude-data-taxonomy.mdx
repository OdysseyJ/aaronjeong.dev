## 어떤 데이터를 측정해야 할까?
- 모든 데이터를 다 측정하려고 하지 마라. (목적을 가져라)
    - 마이페이지 개선
    - 구매전환률 높이기
    - 비즈니스에 더 큰 영향을 주는 데이터를 측정한다.

      (구매전환률 > 마이페이지)

    - 데이터로 어떤 액션을 취하려고 하는지 생각하자. 분석의도를 항상 생각하자.

      (실제로 필요한 데이터인지, 어떤 데이터를 어떻게 수집해야 할 지 명확해진다.)

        예를들어, 스크롤을 어디까지 내리는 지를 측정하고 싶다

        -> 마지막까지 내리는 유저는 제품을 많이 궁금해하고 제품 전환률이 높을거다라는 가정을 가지고 마지막 까지 스크롤을 내리는 이벤트만 수집하기.

## 어떤 지표를 측정해야 할 지 모르는 경우, 대화식으로 알고 싶은 것을 정리하기
- 얼마나 많은 사람들이 어떤 수단으로 (이메일, 네이버, 카카오) 회원 가입을 완료하는지.

  (회원가입 전환률)

- 메인 페이지의 배너 중 어떤 배너가 가장 많은 클릭이 일어나는 지

  (메인배너 CTR)

## Critical Path : Funnel 생각하기
1. 핵심이벤트 정의
2. 시작이벤트 정의
3. 중간이벤트 정의
4. 전체 이벤트의 공통 속성 정의
5. 각 이벤트의 개별 속성 정의

`예시`
- 회원 로그인/가입 화면 -> 로그인 / 가입 버튼 클릭 -> 로그인 / 가입 완료
- 상품 찾기 -> 상품 보기 -> 주문정보 입력 -> 주문 완료

## Product Engagement는 나중에 해도 된다.
- 마이페이지, 햄버거 메뉴, 각 주요 메뉴 메인 페이지 등

## Tips:
- 네이밍 컨벤션, 한글 값 사용, Display Name 사용법, 이벤트명은 사용자의 행동을 가급적
  명확히 표현하기.
    - 왠만하면 가급적 영문, 소문자로 이벤트 정의
    - 네이밍은 무조건 일관성이 있어야 한다.
    - 이벤트 네이밍은 유저의 행동을 정확히 묘사해야 한다 (login -> click login button)

- event : view_page
 
  props : page_name

  value : main, product, category, brand

  -> event : view_main_page, view_product_page, view_category_page, view_brand_page
와 같이 각각의 뷰의 특성이 다를 수 있으므로 이벤트를 분리하는게 낫다.

- 이벤트는 커스텀 이벤트로 정의할 수 있으므로 쪼개는게 더 낫다.
- Revenue(수익) 이벤트는 Revenue 이벤트 전용 메써드나 Revenue Properties를 사용하자.
  (Historical Count관련 프로퍼티 사용 권장 중)
- 여러 상품에 대한 주문 정보가 한 건의 주문에 한꺼번에 전송되는 경우 반복문을 통한
  처리를 권장. 주문당으로 묶어서 봐야 할 경우 Revenue 이벤트와는 별도로 일반 이벤트를 전송하는 방법 권장.
